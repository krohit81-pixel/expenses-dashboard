# API Design

> The original target below was a REST-style `/api/v1` BFF contract. The
> app actually built almost everything as **Next.js Server Actions**
> co-located in each feature (`src/features/<feature>/api/actions.ts`),
> called directly from Server/Client Components — not a separate HTTP API
> layer. The one real route-handler surface is attachments (signed
> upload/download URLs), because that needed a stable HTTP endpoint for
> direct browser upload. This doc is corrected to describe what's actually
> there; the validation/error/money conventions below still hold either
> way.

## Actual mutation pattern

- Every feature's mutations live in `src/features/<feature>/api/actions.ts`
  as `"use server"` functions, called directly from components — no
  hand-rolled fetch/JSON contract to keep in sync.
- Each action validates its input with Zod, calls into the matching
  `src/services/*Service.ts` for business logic, and returns a typed
  result (often a discriminated union for success/validation-error/
  domain-error, e.g. statement import's reconciliation failure vs. parse
  failure vs. success).
- Services never accept a `user_id` from the action — they read
  `OWNER_USER_ID` from `src/lib/owner.ts` directly (see doc 00 — there is
  no session to derive it from).
- The one route-handler surface: `src/app/api/attachments/` issues signed
  Supabase Storage upload/download URLs. Route handlers are the right
  tool specifically because the browser needs a stable URL to `PUT`/`GET`
  against directly, not a server action's request/response.

## Conventions (still accurate)

- Amounts: the branded `Money` decimal-string type (e.g. `"1250.00"`),
  never JSON/JS numbers — see doc 08 and `src/lib/money`.
- Dates: ISO `YYYY-MM-DD`. Timestamps: ISO-8601 UTC (`timestamptz`).
- IDs: UUIDs, generated by Postgres (`gen_random_uuid()`), never
  client-supplied.
- Validate every external boundary (form input, file upload, env vars)
  with Zod before it reaches a service.
- A statement import's discriminated result shape (see
  `CreditCardStatementService`/`StatementImportService`) is the reference
  pattern for "this can fail in several distinct, user-meaningful ways" —
  parse failure, password required/incorrect, reconciliation failure, or
  success — each surfaced with a specific message on the Imports page
  rather than a generic error.

## Error handling

Actions catch known error classes (e.g. `AxisReconciliationError`,
`PdfPasswordRequiredError`, `HdfcHeaderParseError`) and translate them into
a specific user-facing message; anything unexpected propagates as a
generic failure rather than being silently swallowed. There is no
formal `400`/`401`/`403` HTTP status taxonomy in practice, since almost
nothing is a route handler — the discriminated-union return type from each
action is the actual "error class" surface or use.
